Purpose of this Guide
ℹ️
Rapid Cortex is an assistive operational platform. It is intended to support dispatch and response workflows — it does not replace trained personnel, agency policy, or required human judgment. All final decisions remain with your staff.

This guide is for county or city IT administrators, municipal public safety technology teams, 911 center technology support staff, dispatch center supervisors, agency Rapid Cortex administrators, and approved implementation partners.

It explains how to prepare agency workstations, install and configure access, set up users and roles, verify system readiness, maintain the system, respond to common issues, and escalate support when needed.

📋

🔧 Agency Technical Lead
Responsible for workstation readiness, browser and device support, network and firewall rules, user account coordination, and first-line troubleshooting.
📻 Agency Operations Lead
Responsible for dispatch workflow alignment, supervisor escalation procedures, training coordination, and policy and protocol alignment.
🔑 Agency Rapid Cortex Admin
Responsible for user provisioning, role assignment, access review, audit review, and local issue reporting.

Network & Security Prerequisites
Verify these before any user attempts to sign in
🚨Agencies using allowlisting or SSL inspection must resolve these items before go-live. Failure to do so is the most common cause of blank dashboards and auth failures.
  • Outbound HTTPS (port 443) — Allow from dispatch workstations to rapidcortex.us and *.amazonaws.com.
  • Authentication endpoints — Allow outbound access to Cognito authentication domains (provided by your Rapid Cortex rep at setup).
  • WebSocket traffic — If firewall rules inspect or block WebSocket (WSS) connections, whitelist Rapid Cortex API domains. Required if transcript streaming is enabled.
  • SSL inspection / TLS intercept — If your agency performs TLS inspection, add Rapid Cortex and AWS domains to the bypass list. Inspection can break authentication token handling and cause silent login failures.
  • Endpoint protection — Confirm your EDR or antivirus solution does not quarantine browser sessions or flag Rapid Cortex domains. Add to trusted sites if required.
  • Cookies and local storage — Ensure browser policy does not block cookies or local storage for rapidcortex.us. These are required for the login session.
  • Pop-up blockers — Allow pop-ups from rapidcortex.us. Some auth flows (password reset) require them.
  • Workstation clock sync — Ensure workstation time and date are correct. Authentication tokens are time-sensitive — clock drift of more than a few minutes causes silent auth failures.

Initial Local Setup Process
Five steps to verify every new workstation before staff use
✓The Rapid Cortex implementation team handles all cloud, server, and authentication infrastructure. Your IT team's job here is to verify access from each workstation and confirm the system is operational before staff use it.
1
Confirm workstation readiness
Verify: browser or desktop app is installed and updated · workstation date/time is correct · audio devices connected if required · network access is stable · user has approved credentials ready.
2
Launch Rapid Cortex
Open the approved agency URL in Chrome or Edge: https://www.rapidcortex.us/[your-slug]/login — or launch the approved desktop build from the workstation. The login page should load without errors or security warnings.
3
Sign in
Enter assigned credentials. Complete MFA if enabled for your agency. After login, verify the correct agency name and environment are shown in the interface — not a different agency's tenant.
4
Confirm role-based access
Verify the user sees the correct views for their role: dispatchers should see the live dashboard · supervisors should see the review queue · admins should see admin controls · auditors should see only approved read-only views.
5
Verify the system is operational
Confirm: dashboard loads without blank screens · incident data loads · transcript and AI panels load if enabled · no red banners or "API offline" errors appear. If any check fails, do not proceed — consult Chapter 11.
Your agency URL structure
# Sign-in https://www.rapidcortex.us/[your-slug]/login # Live dispatcher dashboard https://www.rapidcortex.us/[your-slug]/dashboard # Supervisor review queue https://www.rapidcortex.us/[your-slug]/review # Admin area (admin role required) https://www.rapidcortex.us/[your-slug]/admin # Training / demo mode ONLY — never use on live calls https://www.rapidcortex.us/[your-slug]/demo
⚠️Replace [your-slug] with your actual jurisdiction slug (e.g. columbus, franklin-county). Bookmark the correct URLs on every workstation before go-live. Never use /demo on live operational calls.

Creating a new user (admin steps)
1
Sign in as admin
Navigate to /[slug]/admin/users using your admin account.
2
Invite the user
Enter the staff member's work email address, select their role, and submit. The system logs the invite in the audit trail automatically.
3
Deliver credentials securely
Credentials are delivered out-of-band during the pilot phase. Follow your agency's policy for secure credential distribution — do not send passwords over unencrypted email.
4
First login — password set
On first login, Cognito prompts the staff member to set a new password. This is automatic. Ensure they complete this before their first operational shift.
5
Verify role and dashboard
Confirm the user lands on the correct dashboard with no errors. If they see wrong-agency content or a blank screen, contact your Rapid Cortex rep — a claim may need correction.
Account changes
✚ New hire
Create the account in Admin → Users before their first shift. Deliver credentials per your agency's policy.
↔ Role change
Update the role in Admin → Users. Changes take effect at next login — ask the user to sign out and back in.
✕ Departure
Deactivate the account on their last day. Do not delete — deactivated records preserve the audit trail for that user's activity.

Admin Configuration Review
Tasks the agency admin should complete during rollout and verify regularly
User provisioning checklist
  • All authorized users created or invited with correct work email addresses.
  • Each user assigned the correct role — confirm with their supervisor if unsure.
  • Every user has signed in at least once and confirmed they reach their correct dashboard.
  • Unused or duplicate accounts disabled before go-live.
Role verification checklist
  • Dispatchers do not have admin functions visible unless explicitly approved.
  • Supervisors can access the review queue and required oversight views.
  • Only approved staff have the admin role — review the list with your operations lead.
Local configuration review
  • Agency name and tenant ID match what your Rapid Cortex rep confirmed during setup.
  • Language and transcript options are configured correctly for your pilot scope.
  • Only approved features are enabled — if something appears that was not expected, contact your rep.
  • Support contacts are documented locally and accessible to all staff (see Appendix B).

  • Network access verified — firewall and allowlist rules confirmed per Chapter 2.

  • Admin can sign in and reach Admin → Users without errors.
  • At least one dispatcher can sign in and sees the live dashboard.
  • At least one supervisor can sign in and sees the review queue.

  • AI analysis works if enabled — at least one test analysis completed.

  • Demo mode is not visible on live-use workstations.
Operations readiness
  • Support contact list posted at every dispatch console (see Appendix B and Wall Card).
  • Escalation path understood by all staff — who to call for what type of issue.
  • Known Limitations reviewed — all staff understand what is and is not active for this agency.
  • Training completed — all operators have completed required Rapid Cortex training.

Daily Operational Startup Check
Run by the designated supervisor or IT lead at the start of each shift
ℹ️This check takes under 5 minutes. If any item fails, consult Chapter 11 before dispatchers use the platform on live calls.
  • Rapid Cortex launches successfully on a test workstation.
  • At least one user can sign in without errors.
  • Dashboard loads current incident data — list is not blank.
  • At least one incident can be opened and viewed.
  • Transcript and AI panels are visible if part of the deployment.
  • No active outage notice is displayed in the interface.
  • Audio devices are functioning if transcript/voice features are enabled.
  • API health indicator in the top bar shows green / Live — not "offline" or error.

Routine Maintenance
Daily through quarterly responsibilities for IT admin and supervisors
✓Rapid Cortex is cloud-hosted — no server patching, no database backups, no software installs on workstations. Your maintenance is primarily user management and access hygiene.
Daily
  • Verify users can access the system (daily startup check)
  • Review any visible error banners reported by dispatchers
  • Confirm no failed login spikes or role issues reported at shift handoff
Weekly
  • Review inactive accounts — deactivate if staff have changed
  • Verify current staff still have correct roles
  • Review any known issues or release notices from Rapid Cortex
  • Confirm browser/workstation updates have not broken access
Monthly
  • Full audit of user list — disable accounts for departed or transferred personnel
  • Review system health dashboard if available
  • Verify support contacts and escalation roster are current
  • Review agency-specific settings for correctness
  • Review privileged roles — confirm no unnecessary admin assignments
Quarterly
  • Formal access review — each user's role and need confirmed
  • Confirm policy alignment — any SOP changes need to be reflected in training
  • Review training refresh needs for staff
  • Validate backup and support contacts are current
Account maintenance rules
  • Remove or disable separated employees promptly on their last day — do not wait for the monthly audit.
  • No shared accounts — every user must have unique credentials unless explicitly approved in writing.
  • Document who the current agency admins are and keep this list updated.

Maintenance & Patching
How updates work — browser-based, desktop, and backend
🌐 Browser-based
Keep approved browsers updated. Validate changes on a test workstation before widespread rollout. Clear cached application data if instructed during support. Hard refresh (Ctrl+Shift+R) resolves most post-update display issues.
🖥️ Desktop app
Install only approved signed versions. Remove obsolete builds. Apply updates during approved maintenance windows. Validate startup and sign-in after every update before releasing to dispatch staff.
☁️ Backend / service
Handled by the Rapid Cortex platform team. Your role: review maintenance notifications, coordinate approved testing windows, and verify system access after any release.
ℹ️Platform updates deploy automatically. Rapid Cortex will notify you in advance of significant changes. If dispatchers report that the interface looks different after an update, a hard refresh resolves most cases.

Common Issues & Fixes
Check → Try → Escalate for each problem type
💡For every issue you escalate: collect UTC time, incident ID, request ID (from any error box), browser version, and whether one or multiple workstations are affected. See Chapter 13 for the full collection checklist.
LoginUser cannot log in
Check
  • Username/password correctness — check CAPS LOCK
  • MFA prompt — is the user completing the second factor?
  • Correct environment URL — www.rapidcortex.us/[slug]/login
  • Browser cache or stale session
  • Account disabled or role removed by admin
  • Identity provider outage (affects all users simultaneously)
Try
  • Sign out fully and relaunch the browser
  • Open a private/incognito window
  • Clear browser cache and cookies for the site
  • Test with another approved browser
  • Use "Forgot password" link — reset email arrives within a few minutes (check spam)
  • Admin triggers manual reset from Admin → Users
Escalate if
  • Multiple users cannot log in simultaneously
  • Auth callback fails repeatedly even after cache clear
  • Agency admin cannot reset or verify account state
ConnectionDashboard is blank after login / "API offline"
Check
  • Network connectivity — can the workstation reach other internet sites?
  • API health indicator in the top bar — is it showing red/error?
  • Expired session — try signing out and back in
  • Firewall or content filter blocking API traffic
  • Environment mismatch — is the user on the right agency URL?
Try
  • Hard refresh: Ctrl+Shift+R (Windows) / Cmd+Shift+R (Mac)
  • Sign out and back in
  • Test on a different workstation — if it works there, the issue is local
  • Verify no content filter or firewall rule was recently updated
Escalate if
  • Blank dashboard affects multiple users across the agency
  • API requests fail consistently even after reconnecting
TranscriptTranscript is not appearing
Check
  • Transcript feature is enabled for your agency (check Admin → Configuration)
  • Audio input source is available and recognized by the browser/app
  • Session is attached to the correct incident or call
  • Microphone permissions are granted for the site
Try
  • Refresh the incident/session view
  • Open a different known-good incident
  • Verify microphone permissions in browser settings
  • Confirm there is active source audio or an inbound transcript feed
Escalate if
  • All transcript sessions fail across the agency
  • Only partial segments appear repeatedly
  • Transcript ingest is delayed well beyond expected thresholds
TranslationTranslation to English is not appearing
Check
  • Multilingual feature is enabled for your agency
  • Source language is in the supported language list
  • System is actually receiving source transcript text (original must appear first)
  • Session has not been marked "low confidence" for language detection
Try
  • Confirm original transcript text appears in the panel
  • Refresh the incident/session panel
  • Test with a supported language sample if in a training environment
Escalate if
  • Original transcript appears but English translation never populates
  • Translation failure affects multiple calls or users
AI PanelAI analysis is missing or delayed
Check
  • Incident contains transcript content — analysis requires source text
  • AI analysis feature is enabled for your agency
  • No active outage banner or backlog indicator visible
  • Session has enough transcript data to trigger analysis
Try
  • Refresh the incident detail page
  • Verify transcript segments exist before requesting analysis
  • Click "Refresh AI" once — do not click repeatedly
  • Confirm the user has permission to view analysis results for their role
Escalate if
  • Analysis fails across multiple incidents consistently
  • Summaries never populate even with transcript present
  • AI error indicators persist after refresh
AccessUser sees "access denied" or missing features
Check
  • Assigned role in Admin → Users
  • Tenant/agency mapping — is the account in the right agency?
  • Whether the feature is restricted to supervisors or admins only
  • Whether the user is on the correct environment URL
Try
  • Admin reviews the user account in Admin → Users
  • User signs out and back in after any role change — claims update at login
  • Verify the account belongs to the correct agency, not a test agency
Escalate if
  • Role assignments appear correct but access is still blocked
  • Multiple users are mapped to the wrong tenant
PerformanceSystem is slow or unresponsive
Check
  • Local workstation performance — CPU and memory usage in Task Manager
  • Browser resource usage — too many tabs or extensions consuming memory
  • Network latency — run a speed test from the workstation
  • Background downloads or patching running on the workstation
Try
  • Close unnecessary tabs and background apps
  • Relaunch the browser or desktop app
  • Test on another workstation — if faster, the issue is local hardware/network
  • Verify local internet and WAN performance
Escalate if
  • Performance degradation affects multiple users across multiple workstations
  • API or transcript latency is consistently high across the agency
BrowserPage looks broken or buttons don't work
Try first
  • Hard refresh: Ctrl+Shift+R / Cmd+Shift+R
  • Clear browser cache: Chrome → Settings → Privacy → Clear cached images/files
  • Open a private/incognito window — if it works there, the issue is a cached file
  • Confirm browser is Chrome 110+ or Edge 110+ — older browsers are not supported
  • Disable browser extensions for rapidcortex.us — ad blockers and script blockers can break UI components

Low-Confidence & Interpreter-Review Handling
What to do when the system flags uncertain output
🚨
Rapid Cortex flags are operational aids — not final authority. When the system marks a transcript, translation, or language detection result as low confidence, treat it as a signal to apply human judgment, not a confirmed result to act on.
When you see a low-confidence or interpreter-review flag
1
Do not assume the output is fully reliable
Low-confidence flags mean the system's certainty is below threshold. The output may still be useful but should not be acted upon without verification.
2
Follow agency SOP for interpreter or human verification
Your agency's standard operating procedure for interpreter engagement or supervisor review takes precedence. Rapid Cortex does not override those procedures.
3
Notify the supervisor if required by policy
Some agencies require supervisor notification when translation confidence is low. Follow your local policy.
4
Continue operations using human judgment
Never pause or delay emergency response waiting for AI output to improve. Use approved procedures and human staff as the primary source of truth.
5
Document repeated low-confidence issues
If a specific language, scenario, or call type consistently generates low-confidence results, log it for technical review and include it in your next Rapid Cortex support check-in.

Information to collect before opening a support ticket
💡Providing this information upfront cuts resolution time significantly. Collect it before calling or emailing support.

  • Steps already attempted — what you tried from Chapter 11.

Troubleshooting Guide
The operational playbook for Rapid Cortex admin staff, support managers, implementation teams, and technical leads — diagnosing, documenting, resolving, and escalating customer issues.
AudienceAdmin staff · Support leads · Technical specialists
ClassificationInternal — do not distribute to customers
ContextMission-critical public-safety environment
🚨SeverityLevels 📋IntakeChecklist 🔐LoginIssues 🎙️AudioIssues 📖PlaybooksScenarios 💬ScriptsCall support
⚡
Mission-critical mindset required. Rapid Cortex supports public-safety agencies. Delayed action, misconfiguration, audio failure, or incorrect routing may affect emergency operations. Every issue must be treated with urgency, clarity, consistency, and complete documentation.
First five questions on every contact
⏱️ Is it happening now?
Determine whether this is an active, ongoing issue or a past event. This sets triage priority immediately.
🚨 Is live emergency workflow impacted?
If yes — escalate before completing intake. Do not delay triage to gather paperwork.
📍 What is the scope?
One user? One workstation? One site? Entire agency? Scope determines severity and escalation path.
🔄 Is there a workaround?
Can the user switch from desktop to web, or to another workstation? A workaround buys diagnosis time.
Golden rules — non-negotiable
  • 🔍Never guess when a fact can be verified. Check logs, timestamps, and account status before forming a conclusion.
  • ✅Never tell a customer an issue is resolved until testing confirms it. Verify the exact failed action works before closing.
  • 📝Never alter production settings without recording what changed. Every change must be logged with timestamp and reason.
  • 🗂️Never close a ticket without root cause, corrective action, and prevention step. Incomplete closures repeat as future incidents.
  • 🤝Never blame the customer. The system exists to serve them. If they misuse it, that's a training gap — not their fault.
  • 📸Always preserve evidence. Logs, timestamps, screenshots, user details, device details, and reproduction steps before making any change.
  • 📣Always assume the customer needs status updates. Silence feels like abandonment to an agency mid-incident.

Classify every issue before troubleshooting begins
S1
Critical
  • Agency cannot use Rapid Cortex for live operations
  • Core calls cannot be processed
  • Audio/transcription completely unavailable during active use
  • Multiple users or entire center down
  • Both desktop and web fail for same agency
  • Routing, dispatch guidance, or core AI nonfunctional
  • Suspected data loss in operational area
Response
Immediate acknowledgment. Escalate to support lead and technical team now. Begin incident log. Do not wait for full intake before starting triage.
S2
High
  • Major feature unavailable but workaround exists
  • Several users impacted
  • AI outputs degraded or unreliable
  • Login failures affecting a team or location
  • Sync delays affect reporting or operational review
  • Audio is intermittent
S3
Medium
  • One user or one workstation affected
  • Non-critical module malfunctioning
  • Issue is repeatable but not blocking operations
  • Configuration mismatch preventing a role from doing all tasks
S4
Low
  • Informational request
  • Cosmetic issue
  • Customer needs guidance or retraining
  • Minor annoyance without operational impact

Standard Troubleshooting Workflow
Follow this sequence for every issue — don't skip steps
1
Confirm scope
Determine whether the issue is user-specific, workstation-specific, network-specific, site-specific, role-specific, environment-specific, or system-wide. Scope determines severity and escalation path.
2
Confirm reproducibility
Can they show the exact steps? Does it happen every time? Does it happen on another device? Does another user on the same machine see the same thing?
3
Check known conditions
Review: current incidents · planned maintenance · recent releases · recent config changes · known bugs · expired credentials or certificates · local network changes.
4
Run category-specific troubleshooting
Use the relevant issue category in Section 6 of this guide. Do not free-form troubleshoot — the category steps exist to be systematic.
5
Test after each change
After any fix or change, verify: the user can perform the failed action · logs no longer show the same error · audio, sync, or routing works · other connected functions remain intact.
6
Document everything
Symptoms · scope · exact steps taken · outcome of each step · final fix or escalation · follow-up actions. See Section 9 for documentation standards.
6A
Login & Access Issues
Can't sign in, wrong permissions, MFA failures, stale sessions
SymptomsCannot sign in · invalid credentials · wrong dashboard · MFA fails · password reset not received
Check
  • Correct email/username entered
  • Account status (active, pending, suspended)
  • Assigned role matches expected access
  • MFA delivery functioning
  • Device date/time is correct
  • Recent password change, domain migration, or auth update
Try
  • Full sign out and re-login
  • Private/incognito window (web)
  • Clear browser cache/cookies for app domain
  • Fully close and relaunch desktop app
  • Test same account on another device
  • Confirm MFA delivery if enabled
Escalate if
  • Authentication service is down
  • Multiple users locked out without explanation
  • Role or token logic is broken
  • Permission mismatch persists after re-login
6B
Desktop App — Won't Open or Crashes
App fails to launch, freezes, crashes on Windows or macOS
SymptomsWon't launch · crashes immediately · freezes on login · permission errors on macOS · hangs after Windows update
Check
  • OS version and exact app version
  • Update or install just before the failure
  • Workstation meets minimum requirements
  • Installed from correct official package
  • Endpoint protection blocking the executable
  • User has local permissions to run the app
Try
  • Fully quit and retry; then reboot
  • Test another user account on same machine
  • Test same user on another machine
  • Clear local cache/config if approved operation
  • Review local crash logs
  • Reinstall from latest approved build if all else fails
Escalate if
  • Multiple workstations fail after same release
  • Crash logs indicate application defect
  • Reinstall does not resolve
  • Issue correlates with new build or env change
🍎 macOS-specific checks
Security settings allow the app to run · Microphone permissions granted · App not quarantined by Gatekeeper · Accessibility or input permissions approved if required
🪟 Windows-specific checks
Antivirus/EDR not sandboxing the app · Install path and user permissions correct · Microphone enabled in Windows Privacy settings · No new GPO restrictions introduced
6C
Web App — Not Loading or Broken
Blank screen, partial load, buttons unresponsive, stale data
SymptomsBlank screen · spinner never completes · partial dashboard · buttons do nothing · websocket/live updates fail · stale data
Check
  • Browser and version (Chrome 110+, Edge 110+)
  • VPN or restrictive proxy active
  • Backend API and websocket service status
  • Auth session not expired
  • Whether issue affects all users or one
Try
  • Test in another supported browser
  • Test in private/incognito window
  • Disable browser extensions
  • Sign out and back in
  • Test from another network if feasible
  • Capture console errors if technical staff available
Escalate if
  • Multiple agencies report same page failure
  • Dashboard actions fail despite successful login
  • Live update channels are down
  • API endpoints show widespread failure
6D
Audio / Microphone / Headset Issues
No audio input, wrong device, distortion, intermittent drops
💡Quick isolation logic: Mic fails in all apps → device or OS issue. Mic works in OS but not Rapid Cortex → app permission or device selection. Audio present but transcription fails → AI/audio pipeline issue.
SymptomsNo audio input · meter flat · transcription blank or incomplete · distorted/intermittent audio · wrong device selected · audio drops mid-session
Check
  • Which headset/mic should be active
  • Correct device selected in app and OS
  • OS microphone permissions
  • Microphone muted or gain too low
  • Competing software locking the device
  • Bandwidth stability if audio is streamed live
Try
  • Select the correct device in app
  • Test device in another application
  • Try another approved headset
  • Disconnect and reconnect the device
  • Reboot the workstation
  • Compare behavior on web vs desktop
Escalate if
  • Multiple users lose audio after an update
  • Audio stream reaches system but transcription doesn't begin
  • Audio devices randomly switch during active workflows
  • Agency-wide audio degradation reported
6E
Transcription Issues
No transcript, late start, omissions, wrong language, lag
SymptomsNo transcript · late start · phrases omitted · wrong language · significant lag · inconsistent quality
Check
  • Audio input is actually present
  • All calls or specific calls only
  • Detected language is correct
  • Expected language configuration is enabled
  • Network stability during the session
  • Speech-to-text provider health
Try
  • Confirm latency vs. accuracy vs. absence
  • Compare performance on another workstation
  • Compare on another call
  • Capture sample audio and output for engineering if quality is poor
Escalate if
  • Transcript service entirely unavailable
  • Latency exceeds operational threshold across many users
  • Language detection consistently fails after a release
  • Missing segments appear across multiple sessions
6F
AI Guidance & Response Quality
No guidance, delays, irrelevant output, incomplete summaries
⚠️Do not tell a customer an AI quality issue is "just user error." Distinguish clearly between: system not responding · system responding slowly · system responding inaccurately. These have different root causes and different escalation paths.
SymptomsNo guidance · delayed output · irrelevant or low-quality response · incomplete summaries · inconsistent confidence
Check
  • Transcription completed correctly first
  • Recent changes to AI workflows or prompts
  • Issue isolated to one model path or all model paths
  • Fallback model behavior occurred
  • One call or many affected
Try
  • Capture exact timestamp and example output
  • Compare same workflow on another session
  • Route examples with full context to engineering
Escalate if
  • AI guidance disappears across agencies or sites
  • Output becomes clearly unreliable in core workflow
  • Output contradicts platform rules or policy
  • Issue began immediately after AI workflow changes
6G

Major Incident Process
Invoke when S1 is declared or multiple customers are affected
🚨Invoke this process for: S1 declaration · multiple customers affected · core operations impaired · production release causing widespread failure · suspected data loss, security exposure, or routing failure.
1
Open incident record immediately
Capture first report time. Do not wait to gather all facts before opening.
2
Identify impacted scope
Which customers, modules, and environments are affected. Stop duplicate troubleshooting from fragmenting information — one owner, one thread.
3
Notify support lead and technical incident owner
Immediate notification regardless of time. The support lead coordinates response; they do not troubleshoot.
4
Send customer-facing update (use template below)
What happened · What is affected · What is not affected · What is being done now · Current workaround · Next update time.
5
Record every change, finding, and time
The incident log is the authoritative record. Every action logged in sequence with timestamps.
6
Confirm resolution with testing before closure
Do not close until the exact failed workflow is confirmed working. Complete post-incident review — see Section 14.

Call Scripts for Admin Staff
Use these verbatim or as a starting point for each stage of a support contact
OpeningFirst contact
"Thank you for contacting Rapid Cortex support. I'm going to document the issue and work through this with you. First, I need to confirm whether this is affecting live operations right now, how many users are impacted, and what exact error or behavior you are seeing."
ClarificationGetting precise details
"To make sure we troubleshoot the right issue, can you walk me through the exact steps you took right before the problem happened and tell me the exact time it occurred?"
ScopeDetermining impact
"Is this happening to just you, this workstation, multiple users in your center, or everyone at your site?"
TestingWalking through changes
"I'd like to test one change at a time so we can identify the exact cause. After each step, please try the same action again and tell me whether the behavior changes."
EscalationHanding off
"I've completed the first-line troubleshooting steps and I'm escalating this to our technical team with the details we captured. I'm documenting the exact symptoms, timestamps, and environment so they can move faster."
ResolutionClosing the case
"We've confirmed the issue is resolved. Before we close this, I want to verify the exact workflow now works as expected on your side and document the root cause and fix for future prevention."

Scenario Playbooks
Step-by-step for the most common inbound scenarios
AOne user cannot log in
  1. Confirm email/username is correct
  2. Verify account status in admin controls
  3. Confirm correct role assigned
  4. Have user retry in web and desktop if available
  5. Clear session or try private window
  6. Reset password or unlock if needed
  7. Escalate if correct account status still fails
BDesktop app opens but no transcription starts
  1. Confirm microphone is selected in the app
  2. Verify OS microphone permissions
  3. Test headset in another application
  4. Reconnect device; relaunch app
  5. Compare on another workstation
  6. Escalate if audio is present but transcript never begins
CEntire center reports slow system
  1. Confirm whether issue affects web, desktop, or both
  2. Confirm exact timeframe and whether all users are impacted
  3. Check current incidents and recent releases
  4. Test network conditions — are other systems slow?
  5. Escalate as high-severity performance incident immediately
DUser sees wrong permissions or access
  1. Verify expected role and actual assigned role
  2. Have user sign out and back in
  3. Confirm tenant/agency mapping is correct
  4. Compare another user with the same role
  5. Escalate if role mapping is correct but permissions remain wrong
EWeb and desktop data do not match
  1. Confirm which action was performed and on which platform
  2. Refresh both clients
  3. Re-login if it is safe to do so
  4. Compare backend record if available
  5. Escalate if persistence or live sync is confirmed failing
